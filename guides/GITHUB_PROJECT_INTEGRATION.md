# GitHub Project Integration Guide

**Version:** 1.0
**Purpose:** Integrate GODMODE workflow with GitHub Projects for issue-driven development

---

## Overview

After plan approval in Phase 0, issues are automatically created in GitHub and added to a GitHub Project board. Claude Code can then pick issues from the board or be assigned via @mentions.

---

## Setup (One-Time)

### 1. Authenticate with Project Scope

```bash
# Required for project access
gh auth refresh -s project

# Verify authentication
gh auth status
```

### 2. Create GitHub Project (if needed)

```bash
# Create new project
gh project create --owner OWNER --title "AI Development Board"

# Or link existing project to repo
# Projects are automatically linked when you add repo issues to them
```

### 3. Identify Your Target Project

```bash
# List all projects and note the TITLE for the one you want
gh project list --owner OWNER

# Example output:
# 3  Claude Agent Protocol         open  PVT_kwHOBGF6h84BJbD_
# 2  Small Business Automation     open  PVT_kwHOBGF6h84BJbAN

# IMPORTANT: Use the exact project TITLE in --project flag
# This ensures issues go to the intended project board
# Note: The number shown is for reference only; use the title in commands
```

### 4. Verify Current Repo

```bash
# Check current repository
gh repo view

# View linked projects (JSON format)
gh repo view --json projectsV2
```

## Creating Issues from Plan

### Automated Flow (via /create-issues)

1. **Identify your target project** (one-time setup):
   ```bash
   # List all your projects and note the EXACT title you want
   gh project list --owner OWNER

   # Example output:
   # 3  Claude Agent Protocol         open  PVT_kwHOBGF6h84BJbD_
   # 2  Small Business Automation     open  PVT_kwHOBGF6h84BJbAN
   #
   # Copy the exact TITLE for use in commands below
   ```

2. **Ensure plan is saved** (GODMODE does this automatically):
   ```bash
   # Plans are initially saved to docs/plans/YYYY-MM-DD-type-feature-name-plan.md
   # Example: docs/plans/2026-02-04-standard-user-authentication-plan.md
   ls docs/plans/
   ```

3. **Generate issues** from approved plan:
   ```bash
   # In Claude Code session after plan approval
   /create-issues docs/plans/2026-02-04-standard-user-authentication-plan.md
   ```

4. **Command outputs** markdown for each issue with metadata

5. **Create first issue** using gh CLI with **exact project title**:
   ```bash
   # IMPORTANT: Use the EXACT project title to specify which board
   # Be explicit to ensure issues go to the intended project

   gh issue create \
     --title "Implement User Authentication Service" \
     --body-file issue-1.md \
     --label "type: feature" \
     --label "priority: high" \
     --label "area: backend" \
     --label "flag: security-sensitive" \
     --project "Claude Agent Protocol" \
     --assignee @me  # or leave unassigned for backlog
   ```

   **Important notes:**
   - Use the exact project title (case-sensitive)
   - Auto-add workflows may still add issues to other projects
   - Verify issue placement with: `gh issue view ISSUE_NUMBER --json projectItems`

6. **Rename plan to include issue number:**
   ```bash
   # After first issue created, note the issue number (e.g., #123)
   # Rename plan file to prepend issue number
   mv docs/plans/2026-02-04-standard-user-authentication-plan.md \
      docs/plans/123-2026-02-04-standard-user-authentication-plan.md

   # Update the issue to reference renamed file
   gh issue edit 123 --body "$(cat issue-1.md | sed 's|docs/plans/2026-02-04-standard-user-authentication-plan.md|docs/plans/123-2026-02-04-standard-user-authentication-plan.md|')"
   ```

   **Result:**
   - Plan filename now starts with issue number
   - Creates direct link between plan and GitHub issue
   - All future issues reference: `docs/plans/123-2026-02-04-standard-user-authentication-plan.md`

7. **Commit and push plan to repository:**
   ```bash
   # CRITICAL: Push plan to git so it's available to anyone picking up the issue
   git add docs/plans/123-2026-02-04-standard-user-authentication-plan.md
   git commit -m "docs: Add plan for user authentication (Issue #123)

   Generated plan for user authentication feature.
   Linked to issue #123.

   ðŸ¤– Generated with Claude Code"

   git push origin main  # or your current branch
   ```

   **Why this is critical:**
   - Without this, plan only exists on your local machine
   - Other developers can't see plan when picking up issue
   - You can't access plan in different sessions/machines
   - Issue references plan file path that won't exist in repo
   - Breaks team collaboration on backlog

8. **Create remaining issues** with updated plan reference

### Manual Label Assignment

Labels should be determined by issue characteristics:

| Characteristic | Labels |
|---------------|--------|
| New functionality | `type: feature` |
| Bug fix | `type: bug` |
| Code improvement | `type: enhancement` or `type: refactor` |
| Touches auth/data/APIs | `flag: security-sensitive` |
| Performance requirements | `flag: performance-critical` |
| Breaking API changes | `flag: breaking-change` |
| Plan Phase 1 | `priority: high` |
| Plan Phase 2+ | `priority: medium` or `priority: low` |

## Working with Project Boards

### View Project Board

```bash
# List all projects
gh project list --owner OWNER

# View specific project
gh project view 1 --owner OWNER

# Open project in browser
gh project view 1 --owner OWNER --web
```

### List Issues in Project

```bash
# List all items in project
gh project item-list 1 --owner OWNER

# Filter by status (requires custom fields setup)
gh project item-list 1 --owner OWNER --format json | jq '.items[] | select(.status=="Ready")'
```

### Check Project from Claude Code

You can query the project board during a session:

```bash
# See what's ready to work on
gh project item-list PROJECT_NUMBER --owner OWNER --format json | \
  jq -r '.items[] | select(.status=="Ready") | "Issue #\(.content.number): \(.content.title)"'
```

## Picking Up Existing Issues from Backlog

**Scenario:** You have 10 issues in backlog, want to pick one and start implementing

**Workflow (Entry Point B in GODMODE):**

### Step 1: List Ready Issues

```bash
# View all ready issues in project
gh project item-list PROJECT_NUM --owner OWNER --format json | \
  jq -r '.items[] | "Issue #\(.content.number): \(.content.title) [\(.status.name)]"'

# Example output:
# Issue #45: Implement password hashing [Ready]
# Issue #46: Add user registration endpoint [Ready]
# Issue #47: Create login endpoint [Blocked]
```

### Step 2: Pick an Issue

In Claude Code session:
```
User: "Let's work on issue #45"
```

### Step 3: Load Issue Context

Claude automatically loads:
```bash
gh issue view 45 --json title,body,labels,projectItems
```

Extracts from issue:
- **Title**: What needs to be built
- **Description**: Why and context
- **Acceptance Criteria**: Definition of done
- **Technical Requirements**: How to build it
- **Testing Notes**: What tests are needed
- **Security/Performance Flags**: Check labels
- **Related Issues**: Check dependencies
- **Plan Reference**: Path to original plan (e.g., `docs/plans/123-2026-02-04-standard-user-auth-plan.md`)

### Step 4: Verify Readiness

Claude checks:
- [ ] Issue has clear acceptance criteria
- [ ] No blocking dependencies (related issues are closed)
- [ ] Technical requirements are understood
- [ ] Issue is not already assigned/in-progress

### Step 5: Restate and Confirm

Claude summarizes:
```
"I'll implement issue #45: Implement password hashing

Summary:
- Build bcrypt password hashing service
- hashPassword() and verifyPassword() functions
- Bcrypt rounds = 12
- Handle edge cases: null, empty, unicode

Labels: type: feature, flag: security-sensitive

âœ“ hashPassword() returns bcrypt hash
âœ“ verifyPassword() validates correctly
âœ“ Edge cases tested

```

### Step 6: Setup Git Branch and Assign

Before implementation:

```bash
# 1. Assign issue to yourself
gh issue edit 45 --add-assignee @me

# 2. Create issue-specific branch
git checkout -b issue-45-password-hashing

# 3. Push branch to remote
git push -u origin issue-45-password-hashing

# 4. Comment on issue

```

**Branch naming:**
- Format: `issue-NNN-brief-description`
- Example: `issue-45-password-hashing`
- Lowercase with hyphens

### Step 7: Jump to Phase 1

Claude skips Phase 0 (already planned) and goes straight to Phase 1:
- Implement code
- Run tests
- Security review (if flagged)
- Code review

### Step 8: Implement (Phase 1)

Claude implements the feature following GODMODE Phase 1:
- Write code following standards
- Generate tests (unit, integration, edge cases)
- Security review (if flagged)
- Code self-review

When implementation is complete:

- All context in issue (self-contained)
- Quick context switching
- Visual progress on board
- Can work through backlog systematically

## GODMODE Integration Points

### Entry Point A: Phase 0, Step 6: Create Issues

After plan approval, the flow forks:

**Fork A: Immediate Execution**
1. `/create-issues prd.md`
2. Create issues with `--assignee @me`
3. Pick first issue and continue to Phase 1
4. Work through issues sequentially
5. Update project board status as you progress

**Fork B: Backlog Mode**
1. `/create-issues prd.md`
2. Create issues without assignee
3. Add to project "Ready" column
4. Exit GODMODE workflow
5. Issues live in backlog for later pickup (use Entry Point B to pick up)

### Entry Point B: Pick Existing Issue (Jump to Phase 1)

**When to use:** Later session, picking up issues from backlog created earlier

See detailed workflow in "Picking Up Existing Issues from Backlog" section above.

### Phase 1: Execution from Issues

**Applies to both:**
- Fork A: Working on first issue immediately after creation
- Entry Point B: Picking up existing issue from backlog

When working on an issue:

1. **Load issue context**:
   ```bash
   # View issue details
   gh issue view 123

   # Check related issues
   gh issue view 123 --json body | jq -r '.body' | grep "Related Issues"
   ```

2. **Verify dependencies**:
   - Check "Depends on" section
   - Ensure blocking issues are closed

3. **Follow GODMODE execution**:
   - Implement code
   - Run tests (TEST_STRATEGY.md)
   - Security review (if `flag: security-sensitive`)
   - Code review (AI_CODE_REVIEW.md)

4. **Update status**:
   ```bash
   # Add comment with progress
   gh issue comment 123 --body "Implementation complete. Running tests..."

   # Close when done
   gh issue close 123 --comment "All acceptance criteria met. Tests passing. Security review completed."
   ```

## Project Board Workflow

### Recommended Columns

1. **Backlog** - Issues not yet ready
2. **Ready** - Ready for development (acceptance criteria clear)
3. **In Progress** - Currently being worked on
4. **Review** - Code review / testing
5. **Done** - Completed and verified

### Status Updates

```bash
# Move issue between columns (requires project field setup)
gh project item-edit --project-id PROJECT_ID --id ITEM_ID --field-id FIELD_ID --value "In Progress"
```

## Best Practices

### 1. One Issue = One PR
Each issue should map to a single, focused pull request.

### 2. Label Consistently
Always apply:
- Exactly one `type:` label
- Exactly one `priority:` label
- One or more `area:` labels
- Zero or more `flag:` labels

### 3. Link Issues
Use GitHub keywords in PR descriptions:
```markdown
Closes #123
Related to #124
Depends on #122
```

### 4. Update Status
Keep project board current:
- Move to "In Progress" when starting
- Move to "Review" when code complete
- Close issue when fully done (tests passing, reviewed, merged)

### 5. Reference Checklists
In issue comments, confirm completion:
```markdown
âœ… Security Review Completed
- Reviewed: ~/.claude/checklists/AI_CODE_SECURITY_REVIEW.md
- All OWASP Top 10 items checked
- No vulnerabilities found

âœ… Code Review Completed
- Reviewed: ~/.claude/checklists/AI_CODE_REVIEW.md
- Edge cases tested (null, empty, boundaries)
- Error handling verified
```
